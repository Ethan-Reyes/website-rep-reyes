# TECH.MD — Engineering Manifesto & Refactor Audit

## 📑 Rationale: The "Why" Behind the Refactor

This document outlines the architectural decisions made to transition this portfolio from a student project to a production-ready engineering tool.

### 1. Asynchronous Control & UI Integrity

- **Refactor:** Implemented a timer-clearance mechanism in the `runBootSequence` function.
- **Why:** The original code allowed background timers to continue after the user clicked "Skip." In a professional environment, "leaking" background processes lead to race conditions and memory bloat. This ensures the UI state is clean and predictable.

### 2. Source of Truth Testing

- **Refactor:** Reconfigured **Jest** to import production logic directly via ES6 modules rather than redefining functions in the test file.
- **Why:** Testing a copy of your code isn't true validation. By linking tests to the actual production files, we ensure that the logic being shipped is the logic that was verified.

### 3. Defensive Security (XSS)

- **Refactor:** Deprecated basic Regex sanitization in favor of a `DOMParser` approach and added `sandbox` attributes to Three.js iframes.
- **Why:** Regex is easily bypassed by nested HTML tags. Protecting user data (especially in aviation diagnostics) requires a "Zero-Trust" posture toward any data injected into the DOM.

### 4. Modular Architecture

- **Refactor:** Removed global scope pollution by moving variables into encapsulated functions and removing `window` attachments.
- **Why:** Global variables risk naming collisions. Professional software architecture relies on encapsulation to ensure different parts of the system can scale without breaking one another.

## 🏛 Project Architecture

- **Performance:** Uses **IntersectionObserver** for navigation highlighting, which is significantly more efficient than standard scroll listeners.
- **Serverless Backend:** Offloads data processing to **Netlify Functions**, allowing for live stats (Visitor Tracking/Resume Clicks) without the security burden of a traditional database.
- **Data Strategy:** Combines **Python** for rapid ETL (Extract, Transform, Load) tasks with **C#** for high-performance, type-safe processing of FAA aviation datasets.
